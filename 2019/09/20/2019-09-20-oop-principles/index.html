面向对象 六大原则

单一职责原则

定义：应该有且仅有一个原因引起类的变更。There should never be more than one reason for a class to change.

好处：类的复杂性降低，可读性提高，可维护性提高，风险降低

建议：接口一定要做到单一职责，类的设计尽量做到只有一个原因引起变化。

单一职责原则中职责的划分是需要根据具体项目中功能粒度界定。同样类型在不同的项目中粒度大小不一，比如在通讯方式中，手机可以作为一个整体来看，而在手机设计中，手机需要划分通话、短信等。

里氏替换原则

第一种定义，也是最正宗的定义：If for each object o1 of type S there is an object o2 of type T such that for all programs P defined in terms of T,the behavior of P is unchanged when o1 is substituted for o2 then S is a subtype of T.（如果对每一个类型为S的对象o1，都有类型为T的对象o2，使得以T定义的所有程序P在所有的对象o1都代换成o2时，程序P的行为没有发生变化，那么类型S是类型T的子类型。）

第二种定义：Functions that use pointers or references to base classes must be able to use objects of derived classes without knowing it.（所有引用基类的地方必须能透明地使用其子类的对象。）

里式替换原则是定义了一个规范或者契约，简单归纳四个含义：

  1. 子类必须完全实现父类的方法；
  2. 子类可以有自己的个性；
  3. 覆盖或实现父类的方法时输入参数可以被放大；
  4. 覆写或实现父类的方法时输出结果可以被缩小；

采用里式替换原则，父类有方法实现时，把父类写为抽象类，这样子类就没有需要替换的父类实例。（理解不够继续琢磨。）

依赖倒置原则

定义：High level modules should not depend upon low level modules.Both should depend upon abstractions.Abstractions should not depend upon details.Details should depend upon abstractions.

定义中有三种含义：

  1. 高层模块不应该依赖低层模块，两者都应该依赖其抽象；
  2. 抽象不应该依赖细节；
  3. 细节应该依赖抽象；

简单理解，抽象是接口或者抽象类。细节则是具体实现类，是实现接口或者继承抽象类的类。实现类不应该依赖实现类，而是依赖对应抽象。依赖应该发生在细节之间。

参考规则：

  1. 每个类尽量都有接口或抽象类，或者抽象类和接口两者都具备
  2. 变量的表面类型尽量是接口或者是抽象类
  3. 任何类都不应该从具体类派生
  4. 尽量不要覆写基类的方法（里式替换原则？）
  5. 结合里氏替换原则使用

接口隔离原则

第一种定义： Clients should not be forced to depend upon interfaces that they don’t use.（客户端不应该依赖它不需要的接口。）

第二种定义： The dependency of one class to another one should depend on the smallest possible interface.（类间的依赖关系应该建立在最小的接口上。）

建立单一接口，不要建立臃肿庞大的接口

迪米特原则

也称为最少知识原则（Least Knowledge Principle，LKP）

定义：一个对象应该对其他对象有最少的了解

迪米特法则的核心观念就是类间解耦，弱耦合

开闭原则

定义：Software entities like classes,modules and functions should be open for extension but closed for modifications.（一个软件实体如类、模块和函数应该对扩展开放，对修改关闭。）

开闭原则是六大原则的基础。其他原则的中心思想都是为了应对未来的变化时，不修改原有的系统，通过扩展实现，保证系统尽量稳定。

原则是指导思想，尽量从这几个角度思考设计的好与坏。但是还要根据实际的环境决定，如果为了遵循某个原则，增加大量的复杂度，得不偿失。

设计模式

设计模式是对软件设计中普遍存在、反复出现的各种问题所提出的通用解决方案。常见有23中设计模式，分别是：

  • 创建型模式：工厂方法模式、抽象工厂模式、单例模式、建造者模式、原型模式

  • 结构型模式：适配器模式、装饰器模式、代理模式、外观模式、桥接模式、组合模式、享元模式

  • 行为型模式：共十一种：策略模式、模板方法模式、观察者模式、迭代器模式、责任链模式、命令模式、备忘录模式、状态模式、访问者模式、中介者模式、解释器模式

设计模式有几种要素：
模式名称、问题描述、解决方案、效果等。

六大原则是思想，设计模式是方法。优化代码逻辑不如优化设计方法，优化设计方法不如优化思想。
