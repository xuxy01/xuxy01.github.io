创建型模式抽象了实例化的过程。创建型模式在什么被创建，谁创建它，它是怎么被创建的，以及何时创建这些方面，独立于系统。

有些创建型模式是相互竞争的，另外一些情况是互补的。

下面一起研究它们的相似点和差异点。

单例模式

意图

保证一个类仅有一个示例，并提供一个访问它的全局访问点

适用性

  1. 当类只有一个实例，并且可以从一个访问点访问时；
  2. 这个唯一实例应该是通过子类化可扩展的，并且无需改变代码就能使用扩展的实例；

结构图

工厂模式

意图

定义一个用于创建对象的接口，让子类决定实例化哪一个类。使类的实例化延迟到子类。

适用性

  1. 当一个类不知道它所必须要实例化的类时；
  2. 当一个类希望由他的子类来指定实例化的对象时；
  3. 将创建子类的职责信息局部化时；

结构图

相关应用

  1. Creator类是抽象类，缺少缺省实现；或者Creator类是具体类，包含缺省实现；
  2. 参数化工厂方法；

抽象工厂模式

意图

提供一个创建一系列相关或者相互依赖对象的接口，而无需指定他们具体的类

适用性

  1. 系统要独立于它的产品的创建、组合和表示时；
  2. 系统要有多个产品系列中的一个来配置时；
  3. 强调一系列相关的产品对象的设计，以便联合使用时；
  4. 一个产品类库，只想显示接口而不是实现时；

结构图

相关应用

  1. 将工程座位单例；
  2. 通过工厂方法或者原形模式创建产品；
  3. 通过参数定义可扩展的工厂；

建造者模式

意图

将一个复杂对象的构建与它的表示分离，使同样的构建过程可以创建不同的表示。

适用性

  1. 创建复杂对象的算法应该独立于该对象的组成以及装配方式时；
  2. 构建过程必须允许被构造的对象有不同的表示时；

结构图

相关应用

  1. builder类接口必须足够通用；
  2. 产品没有抽象类，因为生成产品相差可能会很大，不需要公共的父类；
  3. builder中的方法都为空，而不是声明为虚函数。这样可以只重定义需要的操作生成产品；

原型模式

意图

用原型实例指定创建对象的种类，并且通过拷贝这些原型创建新的对象。

适用性

  1. 当要实例化的类是在运行时指定；
  2. 为避免创建一个与产品类层级相同的工厂类；
  3. 一个类的实例只能有几种不同状态组合中的一种时；

结构图

相关应用

  1. 运行时增加或者删除产品；
  2. 改变一个对象变量或者结构，以指定新的对象；
  3. 减少子类的构造；
  4. 用类动态配置应用；

总结

设计通常以工厂模式开始，当发现需要更灵活的设计时，便会考虑其他创建型模式。了解多个模式，以便需要权衡设计时提供更多的选择余地。
